## Week 7 - Ansible

### what is ansible
- ansible is a configuration management tool
	- can define tasks and settings to be applied or run on a system instead of manually
- only requires python and no extra software
- opens SSH connection to the target host
- install with `sudo apt install ansible`

#### ansible concepts
**control node**
- system that ansible is installed
- run commands like `ansible` or `ansible-invnentory`
	- inspects and visualized the inventory of hosts that ansible is currently tracking

**managed node**
- remote system or host that ansible controls
- individual server manage by ansible

**`ansible.cfg`**
- configuration file used by ansible to customize various settings and behaviours of ansible operations
- control how ansible interacts with managed hosts, how to handle SSH connection, and other default behaviours 

```
[defaults]
inventory = ./inventory/hosts.yml
remote_user = admin
private_key_file = ~/.ssh/aws
[ssh_connection]
host_key_checking=False
ssh_common_args= -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null
ansible_ssh_common_args= -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null
```

**inventory**
- collection of "hosts" (machines or artifacts) that can be provisioned using ansible
- can be defined statically (inventory is a file), or generated by a program dynamically
- in static inventory, host can be specified by DNS name, SSH alias, or IP
- inventory can be in INI or YAML format

```ini
[webservice]
httpsrv
dbsrv
appsrv
```

### modules and plugins
- plugin extend ansible fore functionality
	- most plugin types execute on the control note within the `/usr/bin/ansible process`
- modules are type of plugin that execute automation tasks on a "target" (usually a remote system)
	- work as standalone scripts that ansible executes in their own process outside of the control node
- an ansible module is a piece of code that executes a specific provisioning task on a manage host 
	- examples:
		- `file` to provision a file
		- `shell` to run a shell command
		- `get_url` to download a file
		- `package` to work with the package manager from the OS
- ansible modules are divided into core modules (handled in ansible) and community modules (available as a separate package)
- `ansible-doc` is like man pages for ansible

### ad-hoc commands
- ansible ad-hoc commands are one of the commands executed in the command line instead of in a file
- useful for one off commands like rebooting a server

```
ansible \ 
        <pattern> \ 
        <ansible arguments> \ 
        -m <module> \ 
        -a "<module arguments>"
```

### playbooks
- playbooks are the files that define how ansible modules should be applied to the hosts in the inventory
- written using ansible grammar in YAML
- human readable 

#### validating your ansible playbooks
- `yamllint` 
- `sudo apt install yamllint`

#### running a playbook
```
ansible/
    ansible.cfg
    playbook.yml
    inventory/
		hosts.yml
```
- run your playbook with the command `ansible-playbook playbook.yml`

### variables
- playbooks can contain veriables
- variables can be defined in your inventory, playbook, or in separate variable files\

```
---
- hosts: all
  remote_user: root
  vars:
    favcolor: blue
  vars_files:
    - /vars/external_vars.yml
  tasks:
    - name: This is just a placeholder
      ansible.builtin.command: /bin/echo foo
```
- variables are referenced with jinja2 syntax ``dest: "{{ remote_install_path }}/foo.cfg"``
- when you use a variable that contains additional scalar values, you need to wrap the entire value in quotes

## week 9 - terraform 

### terraform input variables 
- input variables are parameters that allow users to send information into a module
	- they make modules reusable and customizable without changing their source code
- key points
	- defined using the `variable` block
	- accessed inside a module using the `var.<variable name>` syntax
	- can include descriptions, type constraints, default values, and sensitive flags 
	- like the function arguments in a programming language such as pytho

```
variable "instance_type" {
  type        = string
  description = "The type of EC2 instance to launch"
  default     = "t3.micro"
}
```

### terraform output variables
- output variables are used to return information from a module to its caller (another module or the root configuration)
- key points
	- defined using the `output` block 
	- typically used to share resource attribute (like instance ID or IP address)
	- can also be marked as sensitive to hide values in logs
	- like a function's return value - send data out of the module
```
output "instance_ip" {
  description = "Private IP address of the instance"
  value       = aws_instance.example.private_ip
}
```

### terraform local variables
- local variables are values that exist only inside a module
	- used for intermediate calculations 
- key points
	- defined inside one or more `locals {}` blocks
	- accessed using the `local.<variable name>` syntax
	- can reference other locals or inputs but cannot be accessed outside the module 
	- line internal variables within a function
		- used for internal logic and not exposed externally
```
locals {
  common_tags = {
    ManagedBy = "Terraform"
    Environment = var.environment
  }
}
```


|Terraform Variable Type|Purpose|Analogy in Programming|Accessibility|
|---|---|---|---|
|**Input Variable**|Pass data _into_ a module|Function **argument**|From outside the module|
|**Local Variable**|Store internal or reusable values|Function **local variable**|Inside the module only|
|**Output Variable**|Return data _from_ a module|Function **return value**|Accessible from outside the module|

### intro to terraform modules
- technically everything in terraform is a module
- every terraform project has at least one module, the root module
- a module that has been called by another module is often called a child module
- modules allow you to write reusable code that you can store locally or remotely
- modules can be used to create infrastructure, or deployment states, or for a host of other scenarios where it's good to have separate reusable configuration
- a module contains terraform configuration
	- could have main.tf, variables.tf, and outputs.tf in a module 

#### where can modules be stored?
- locally in a directory on your machine
- modules can be stored on the terraform registry and the terraform cloud
- remote git repository (or any other version control)
- in S3 bucket
- can host yor own at an HTTP end point

#### what is the difference between resources and modules?
- resource in terraform described a piece of infrastructure to be created (ec2, VPC, subnet)
- module is a collection of resources that are used together to achieve a reusable use case

#### building a module
- a terraform module is a collection of standard configuration files in a dedicated directory
- encapsulates groups of resources dedicated to one task
	- reduces amount of code you have to develop for similar infrastructure components 
```
.
├── README.md
├── main.tf
├── variables.tf
├── outputs.tf
├── ...
├── modules/
│   ├── nestedA/
│   │   ├── README.md
│   │   ├── variables.tf
│   │   ├── main.tf
│   │   ├── outputs.tf
│   ├── nestedB/
│   ├── .../
```

### using a terraform module
- reference a module in a module block
```
module "terraform_test_module" {
 source  = "./module/terraform-test-module"
 version = "1.0.0"
 
 argument_1                     = var.test_1
 argument_2                     = var.test_2
 argument_3                     = var.test_3
}
```

#### passing data into the module with variables
inside the modules variables file
```
variable "instance_type" {
  description = "The type of EC2 Instances to run (e.g. t2.micro)"
  type        = string
}
```

inside the modules main file
```
resource "aws_launch_configuration" "example" {
  image_id        = "ami-0fb653ca2d3203ac1"
  instance_type   = var.instance_type
  security_groups = [aws_security_group.instance.id]
  user_data = templatefile("user-data.sh", {
    server_port = var.server_port
    db_address  = data.terraform_remote_state.db.outputs.address
    db_port     = data.terraform_remote_state.db.outputs.port
  })
  # Required when using a launch configuration with an auto scaling group.
  lifecycle {
    create_before_destroy = true
  }
}
```

in your root main file
```
module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"
  cluster_name           = "webservers-stage"
  db_remote_state_bucket = "(YOUR_BUCKET_NAME)"
  db_remote_state_key    = "stage/data-stores/mysql/terraform.tfstate"
  instance_type = "t2.micro"
  min_size      = 2
  max_size      = 2
}
```

#### accessing data in a module
inside your module code use output variables

```
output "asg_name" {
  value       = aws_autoscaling_group.example.name
  description = "The name of the Auto Scaling Group"
}
```

from your root module, you can access module outputs like
``module.frontend.asg_name``

### terraform module best practices
- each terraform module should live in its own repository and versioning should be leveraged
- minimal structure: main.tf, variables.tf, output.tf
- use input and output variables
	- `module.module_name.output_name`
- used for multiple resources, a single resource module is usually bad practices
- use defaults or optional depending on the data type of your variables 
- use dynamic blocks
- use ternary operators and take advantage of terraform built in functions

## Week 10 - Ansible loops

### ansible with loops
- examples of commonly used loops include changing ownership on several files
	- ansible will automatically replace the variable item with the current value in a loops iteration

e.g. So in the example below the first iteration `item=user-name1` and the second iteration `item=user-name2`.

```
- name: Add several users to the wheel group
  ansible.builtin.user:
    name: "{{ item }}"
    state: present
    groups: "wheel"
  loop:
     - user-name1
     - user-name2
```

#### installing packages with a loop
- some ansible modules accept a list as a parameter value like package managers

```
- name: Optimal yum
  ansible.builtin.yum:
    name: "{{ list_of_packages }}"
    state: present
- name: Non-optimal yum, slower and may cause issues with interdependencies
  ansible.builtin.yum:
    name: "{{ item }}"
    state: present
  loop: "{{ list_of_packages }}"
```

```
# create an array of packages
packages=("vim" "ripgrep" "tmux")
# use a loop to install packages
# calls the "dnf install" command for every package
for package in "${packages[@]}"; do
  dnf install $package
done
# Install all packages with a single "dnf install" command
dnf install "${packages[@]}"
```

### ansible conditionals with when
- may want to execute different tasks or goals depending on the value of a fact, variable, or result of a previous tasks
- may want the value of some variables to depend on the values of other variables

```
- name: Install ngnix
  hosts: all
  become: true
  tasks:
  - name: Get the distribution
    set_fact: 
      distribution: "{{ ansible_facts['distribution'] }}" 
  - name: install nginx in Fedora
    dnf:
      name=nginx
    when: distribution == "Fedora"
  - name: install nginx in Debian
    apt:
      name: ngnix
    when: distribution == "Debian"
```
